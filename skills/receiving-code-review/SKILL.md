# Receiving Code Review（接收代码审查）

## 概述

代码审查需要技术评估，不是情感表演。

**核心原则：实现前验证。假设前询问。技术正确性高于社交舒适。**

---

## 响应模式

```
当收到代码审查反馈时：

1. READ: 完整阅读反馈不反应
2. UNDERSTAND: 用自己的话重述要求（或询问）
3. VERIFY: 对照代码库现实检查
4. EVALUATE: 对此代码库技术上合理？
5. RESPOND: 技术确认或有理反驳
6. IMPLEMENT: 一次一项，每项测试
```

---

## 禁止响应

**绝不：**

- "你完全正确！"（明确违反）
- "好点！" / "优秀反馈！"（表演性）
- "让我现在实现那个"（验证前）

**而是：**

- 重述技术要求
- 问澄清问题
- 如果错了用技术推理反驳
- 直接开始工作（行动 > 话语）

---

## 处理不清楚反馈

```
如果任何项目不清楚：
  停止 - 暂不实现任何东西
  对不清楚项目询问澄清

为什么：项目可能相关。部分理解 = 错误实现。
```

**示例：**

```
用户："修复 1-6"
你理解 1,2,3,6。4,5 不清楚。

❌ 错误：现在实现 1,2,3,6，稍后问 4,5
✅ 正确："我理解项目 1,2,3,6。继续前需要对 4 和 5 澄清。"
```

---

## 来源特定处理

### 来自用户

- **可信** — 理解后实现
- **如果范围不清仍要问**
- **无表演性同意**
- **跳到行动**或技术确认

### 来自外部审查者

```
实现前：
  1. 检查：对此代码库技术上正确？
  2. 检查：破坏现有功能？
  3. 检查：当前实现有原因吗？
  4. 检查：在所有平台/版本工作？
  5. 检查：审查者理解完整上下文？

如果建议看起来错了：
  用技术推理反驳

如果不容易验证：
  这么说："我无法在没有 [X] 的情况下验证。我应该 [调查/询问/继续]？"

如果与用户之前决定冲突：
  停止并先与用户讨论
```

**用户规则：** "外部反馈 — 要怀疑，但仔细检查"

---

## YAGNI 检查"专业"功能

```
如果审查者建议"正确实现"：
  grep 代码库实际使用

  如果未使用："此端点未被调用。移除它（YAGNI）？"
  如果使用：然后正确实现
```

**用户规则：** "你和审查者都向我汇报。如果我们不需要这个功能，不要添加它。"

---

## 实现顺序

```
对于多项反馈：
  1. 首先澄清任何不清楚的
  2. 然后按此顺序实现：
     - 阻塞问题（破坏、安全）
     - 简单修复（拼写错误、导入）
     - 复杂修复（重构、逻辑）
  3. 单独测试每个修复
  4. 验证无回归
```

---

## 何时反驳

当以下情况反驳：

- 建议破坏现有功能
- 审查者缺乏完整上下文
- 违反 YAGNI（未使用功能）
- 对此技术栈技术上不正确
- 存在遗留/兼容性原因
- 与用户架构决策冲突

**如何反驳：**

- 使用技术推理，不是防御性
- 问具体问题
- 引用工作的测试/代码
- 如果是架构问题让用户参与

---

## 确认正确反馈

当反馈确实正确时：

```
✅ "已修复。[简要描述改变了什么]"
✅ "好发现 - [具体问题]。在 [位置] 修复。"
✅ [直接修复并在代码中展示]

❌ "你完全正确！"
❌ "好点！"
❌ "谢谢发现那个！"
❌ "谢谢[任何东西]"
❌ 任何感激表达
```

**为什么不感谢：** 行动说话。直接修复。代码本身显示你听到了反馈。

**如果你发现自己要写"谢谢"：** 删除它。改为陈述修复。

---

## 优雅纠正你的反驳

如果你反驳了但错了：

```
✅ "你是对的 - 我检查了 [X] 它确实 [Y]。现在实现。"
✅ "验证了这个你是正确的。我最初的理解错了因为 [原因]。修复中。"

❌ 长道歉
❌ 为为什么反驳辩护
❌ 过度解释
```

实事求是陈述纠正并继续。

---

## 常见错误

| 错误             | 修复               |
| ---------------- | ------------------ |
| 表演性同意       | 陈述要求或直接行动 |
| 盲目实现         | 先对照代码库验证   |
| 不测试批量       | 一次一个，每项测试 |
| 假设审查者对     | 检查是否破坏东西   |
| 避免反驳         | 技术正确性 > 舒适  |
| 部分实现         | 先澄清所有项目     |
| 无法验证但仍继续 | 陈述限制、询问方向 |

---

## 底线

**外部反馈 = 要评估的建议，不是要遵循的命令。**

验证。质疑。然后实现。

无表演性同意。总是技术严谨。
