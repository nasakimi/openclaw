# Subagent-Driven Development（Subagent 驱动开发）

通过为每个任务派遣新鲜 subagent 执行计划，每个之后有两阶段审查：先规格合规审查，再代码质量审查。

**核心原则：每任务新鲜 subagent + 两阶段审查（先规格再质量）= 高质量、快迭代**

---

## 何时使用

**vs. Executing Plans（并行会话）：**

- 同一会话（无上下文切换）
- 每任务新鲜 subagent（无上下文污染）
- 每任务后两阶段审查：先规格合规，再代码质量
- 更快迭代（任务间无人介入）

---

## 流程

```
读取计划，提取所有任务及完整文本，记录上下文，创建任务列表

[对每个任务]
  ├── 派遣实现者 subagent
  │   └── 如果有问题？回答问题、提供上下文 → 重新派遣
  │   └── 实现、测试、提交、自我审查
  │
  ├── 派遣规格审查者 subagent
  │   └── 规格合规？
  │       └── [否] → 实现者修复 → 重新审查
  │       └── [是] → 继续
  │
  ├── 派遣代码质量审查者 subagent
  │   └── 批准？
  │       └── [否] → 实现者修复 → 重新审查
  │       └── [是] → 标记任务完成
  │
  └── [更多任务?] → 下一个任务

[所有任务后]
  └── 派遣最终代码审查者
  └── 使用 finishing-a-development-branch
```

## 模型选择

使用能处理每个角色的最低能力模型以节省成本并增加速度。

**机械实现任务**（隔离函数、清晰规格、1-2 文件）：使用快速、便宜模型。当计划明确时，大多数实现任务是机械的。

**集成和判断任务**（多文件协调、模式匹配、调试）：使用标准模型。

**架构、设计和审查任务**：使用最强大可用模型。

**任务复杂度信号：**

- 触及 1-2 文件有完整规格 → 便宜模型
- 触及多个文件有集成顾虑 → 标准模型
- 需要设计判断或广泛代码库理解 → 最强大模型

---

## 处理实现者状态

实现者 subagents 报告四种状态之一。适当处理每种：

**DONE:** 继续规格合规审查。

**DONE_WITH_CONCERNS:** 实现者完成了工作但标记了疑虑。继续前阅读担忧。如果担忧关于正确性或范围，审查前解决。如果是观察（如"这个文件变大"），记录并继续审查。

**NEEDS_CONTEXT:** 实现者需要未提供的信息。提供缺失上下文并重新派遣。

**BLOCKED:** 实现者无法完成任务。评估阻塞：

1. 如果是上下文问题，提供更多上下文并用相同模型重新派遣
2. 如果任务需要更多推理，用更强大模型重新派遣
3. 如果任务太大，分解为更小块
4. 如果计划本身错了，升级到人类

**绝不**忽略升级或强制相同模型无更改重试。如果实现者说卡住了，有些东西需要改变。

## 示例工作流

```
你：我正在使用 Subagent-Driven Development 执行此计划。

[读取计划文件一次：docs/plans/feature-plan.md]
[提取所有 5 个任务及完整文本和上下文]
[创建包含所有任务的任务列表]

任务 1：Hook 安装脚本

[获取任务 1 文本和上下文（已提取）]
[派遣实现 subagent 带完整任务文本 + 上下文]

实现者："开始前 - hook 应该安装在用户级还是系统级？"

你："用户级（~/.config/hooks/）"

实现者："明白了。现在实现..."
[稍后] 实现者：
  - 实现了 install-hook 命令
  - 添加了测试，5/5 通过
  - 自我审查：发现漏了 --force 标志，已添加
  - 已提交

[派遣规格合规审查者]
规格审查者：✅ 规格合规 - 所有需求满足，无额外

[获取 git SHAs，派遣代码质量审查者]
代码审查者：优点：测试覆盖好、干净。问题：无。已批准。

[标记任务 1 完成]

任务 2：恢复模式

...（继续相同模式）

[所有任务后]
[派遣最终代码审查者]
最终审查者：所有需求满足，准备合并

完成！
```

## 优势

**vs. 手动执行：**

- Subagents 自然遵循 TDD
- 每任务新鲜上下文（无混淆）
- 并行安全（subagents 不干扰）
- Subagent 可以提问（工作前和工作期间）

**vs. Executing Plans：**

- 同一会话（无交接）
- 持续进展（无等待）
- 自动审查检查点

**效率收益：**

- 无文件读取开销（控制器提供完整文本）
- 控制器精确策划需要什么上下文
- Subagent 提前获得完整信息
- 工作开始前问题浮现（不是之后）

**质量门：**

- 自我审查在交接前捕获问题
- 两阶段审查：规格合规，再代码质量
- 审查循环确保修复实际工作
- 规格合规防止过度/不足构建
- 代码质量确保实现构建良好

---

## 红旗警告

**绝不：**

- 没有明确用户同意在 main/master 分支开始实现
- 跳过审查（规格合规或代码质量）
- 带未修复问题继续
- 并行派遣多个实现 subagents（冲突）
- 让 subagent 读取计划文件（改为提供完整文本）
- 跳过场景设置上下文（subagent 需要理解任务在哪里适应）
- 忽略 subagent 问题（让他们继续前回答）
- 在规格合规上接受"差不多"（规格审查者发现问题 = 没完成）
- 跳过审查循环（审查者发现问题 = 实现者修复 = 再次审查）
- 让实现者自我审查替换实际审查（都需要）
- **规格合规 ✅ 前开始代码质量审查**（错误顺序）
- 任一审查有开放问题时移动到下一个任务

**如果 subagent 提问：**

- 清晰完整回答
- 如果需要提供额外上下文
- 不要急于让他们进入实现

**如果审查者发现问题：**

- 实现者（相同 subagent）修复它们
- 审查者再次审查
- 重复直到批准
- 不要跳过重新审查

**如果 subagent 任务失败：**

- 派遣修复 subagent 带具体指令
- 不要尝试手动修复（上下文污染）

## 集成

**必需工作流 skills：**

- **using-git-worktrees** — REQUIRED: 开始前设置隔离工作区
- **writing-plans** — 创建此 skill 执行的计划
- **requesting-code-review** — 审查者 subagents 的代码审查模板
- **finishing-a-development-branch** — 所有任务后完成开发

**Subagents 应该使用：**

- **test-driven-development** — Subagents 对每个任务遵循 TDD

**替代工作流：**

- **executing-plans** — 用于并行会话而不是同会话执行

## 链式调用

```
[当前 skill: subagent-driven-development]
           ↓
    所有任务完成？
           ↓ [是]
    【必须】调用 verification-before-completion skill
           ↓
    验证通过？
           ↓ [是]
    【必须】调用 finishing-a-development-branch skill
           ↓
    [下一个 skill: finishing-a-development-branch]
```

**完成此 skill 后必须调用下一个 skill：**

1. 先调用 verification-before-completion
2. 验证通过后调用 finishing-a-development-branch
